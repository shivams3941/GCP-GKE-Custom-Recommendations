# GKE Gold Layer — Custom Recommendations

## Overview

This document covers the end-to-end implementation of GKE cost optimization recommendations.
The system is split into two layers:

- **Bronze Layer** — fetches raw GKE cluster, node pool, and metrics data from GCP and stores it
- **Gold Layer** — reads bronze data, evaluates threshold rules, and generates rightsizing recommendations

Recommendation types supported:
- **Idle** — node pool is running but barely used; recommend deleting or scaling to 0
- **Overprovisioned** — node pool is used but oversized; recommend downsizing to a smaller machine type

Install with:

```bash
pip install google-cloud-container google-cloud-monitoring google-auth requests
```

### GCP Service Account Key

The service account key file must be present at:

```
gold-layer/gcp/devops-internal-439011-f4e928045fd3.json
```

The service account needs the following GCP roles:
- `roles/container.viewer` — to list GKE clusters and node pools
- `roles/monitoring.viewer` — to read Cloud Monitoring metrics
- `roles/billing.viewer` — to call the Cloud Billing SKU API (for cr_gke_fetch.py)

To use a different key file, set the environment variable:

```bash
set GOOGLE_APPLICATION_CREDENTIALS=C:\path\to\your-key.json
```

## Step-by-Step: Local Testing

### Step 1 — Generate Bronze Data

Run the bronze-layer local test script. This connects to GCP, fetches real cluster data and metrics, and writes everything to a flat CSV.

```bash
cd bronze-layer
python tests/local_run_gcp_gke.py
```

Output file: `bronze-layer/tests/output/gke_output.csv`

The CSV contains one row per metric data point with all cluster and node pool fields repeated alongside it:

| Column | Description |
|---|---|
| project_id | GCP project ID |
| cluster_name | GKE cluster name |
| location | Zone or region (e.g. us-central1-a) |
| status | Cluster status (RUNNING, etc.) |
| current_node_count | Number of nodes in the node pool |
| machine_type | Node pool machine type (e.g. e2-medium) |
| node_pool_name | Node pool name |
| metric_name | GCP Cloud Monitoring metric name |
| metric_timestamp | Timestamp of the data point |
| metric_value | Metric value (ratio 0-1 for utilization metrics) |

---

### Step 2 — Fetch SKU Pricing

Run the SKU fetch script to pull live machine type pricing from the GCP Cloud Billing API.
This only needs to be re-run when you want fresh pricing or are adding a new region.

```bash
cd gold-layer/gcp/resources/scripts
python cr_gke_fetch.py us-central1
```

Output file: `gold-layer/gcp/resources/skus/cr_gke_sku.json`

The script:
1. Authenticates using the service account key
2. Calls the GCP Cloud Billing Catalog API (`cloudbilling.googleapis.com/v1/services/6F81-5844-456A/skus`)
3. Filters to Compute Engine CPU and RAM SKUs for the given region
4. Builds per-machine-type pricing by multiplying vCPUs x cpu_price + RAM_GB x ram_price
5. Assigns ordinals (lower ordinal = cheaper machine type within the same family)

Supported machine families: `e2`, `n1`, `n2`, `n2d`, `c2`, `c2d`, `t2d`

To fetch pricing for a different region:

```bash
python cr_gke_fetch.py europe-west1
python cr_gke_fetch.py us-east1
```

---

### Step 3 — Run Recommendations

```bash
cd gold-layer/gcp
python _local_dev.py
```

This script:
1. Loads the SKU catalog from `resources/skus/cr_gke_sku.json`
2. Reads the bronze CSV from `bronze-layer/tests/output/gke_output.csv`
3. Aggregates metrics per cluster (averages all data points per metric)
4. Converts GCP ratio values (0-1) to percentages (0-100)
5. Evaluates each cluster against the threshold rules
6. Calls the GKE engine for triggered rules
7. Prints recommendations and savings to stdout

## Metrics Collected (Bronze Layer)

All metrics come from GCP Cloud Monitoring under the `kubernetes.io` namespace.
Values are returned as ratios (0.0 to 1.0) and converted to percentages in the gold layer.

| GCP Metric | Engine Key | Used For |
|---|---|---|
| kubernetes.io/node/cpu/allocatable_utilization | cpu_utilization_avg | Idle + Overprovisioned detection |
| kubernetes.io/node/memory/allocatable_utilization | memory_utilization_avg | Idle + Overprovisioned detection |
| kubernetes.io/container/cpu/request_utilization | container_cpu_avg | Supporting signal |
| kubernetes.io/container/memory/request_utilization | container_memory_avg | Supporting signal |
| kubernetes.io/pod/volume/total_bytes | — | Collected, not used in rules yet |

Metrics are fetched with a 7-day window at 5-minute intervals using `Average` aggregation.

## Recommendation Rules

Rules are defined in `_local_dev.py` as a list of dicts. In production these would be stored in the `finops_threshold_rules_v2` database table.

### GKE_IDLE

Triggered when the node pool is running but barely consuming resources.

| Condition | Operator | Threshold | Logic |
|---|---|---|---|
| cpu_utilization_avg | less than | 5% | AND |
| memory_utilization_avg | less than | 20% | AND |

Action: Delete the node pool or scale it to 0 nodes.
Savings: Full current annual cost of the node pool (no target SKU).

### GKE_OVERPROVISIONED

Triggered when the node pool is active but the machine type is larger than needed.

| Condition | Operator | Threshold | Logic |
|---|---|---|---|
| cpu_utilization_avg | less than | 40% | OR |
| memory_utilization_avg | less than | 50% | OR |

Action: Downsize to a smaller machine type in the same family.
Savings: Difference between current and target annual cost, multiplied by node count.

## Engine Logic (GKEEngine)

Located at: `gold-layer/gcp/rightsize_engine/services/cr_gke_engine.py`

### IDLE path

1. Look up current machine type in the SKU catalog
2. Calculate annual cost = `hourly_price x 8760 x node_count`
3. Return savings = full annual cost (target cost = 0)

### OVERPROVISIONED path

1. Look up current machine type in the SKU catalog
2. Extract machine family (e.g. `e2` from `e2-medium`)
3. Calculate required capacity with a 1.3x safety margin:
   - `required_vcpus = current_vcpus x (cpu_util / 100) x 1.3`
   - `required_ram = current_ram x (mem_util / 100) x 1.3`
4. Find candidates in the same family that:
   - Have a lower ordinal (cheaper)
   - Meet or exceed required vCPUs and RAM
5. Pick the candidate with the fewest vCPUs that still fits
6. If no direct candidate found, fall back to the 3 closest cheaper machine types in the same family
7. Return savings = `(current_annual - target_annual) x node_count`

### Return format

Every engine call returns either `None` or a list:

```python
[
    recommendation_text,      # str  — human-readable recommendation
    current_annual_price,     # float — current yearly cost (USD)
    target_annual_price,      # float — target yearly cost (USD), 0.0 for IDLE
    annual_savings,           # float — yearly savings (USD)
    details,                  # dict  — full breakdown including target SKU specs
]
```

## SKU Catalog Structure

File: `gold-layer/gcp/resources/skus/cr_gke_sku.json`

```json
{
  "e2": {
    "family": "e2",
    "region": "us-central1",
    "cpu_hourly_price": 0.02289,
    "ram_hourly_price_per_gb": 0.003067,
    "machine_types": [
      {
        "machine_type": "e2-medium",
        "family": "e2",
        "vcpus": 1,
        "ram_gb": 4,
        "hourly_price": 0.035158,
        "monthly_price": 25.665,
        "annual_price": 307.98,
        "ordinal": 3
      }
    ]
  }
}
```

Ordinals are assigned within each family by price — ordinal 1 is the cheapest machine type.
The engine uses ordinals to find cheaper candidates without needing to compare raw prices.

## Iceberg Tables (Production)

In production the bronze layer writes to three Iceberg tables on S3 via AWS Glue:

| Table | Description |
|---|---|
| bronze_gcp_gke_clusters | One row per cluster per ingestion run |
| bronze_gcp_gke_node_pools | One row per node pool per ingestion run |
| bronze_gcp_metrics_v2 | One row per metric data point per resource |

All tables are partitioned by `(client_id, account_id, year_month)` and use MERGE INTO upserts keyed on `(client_id, account_id, resource_id)`.

## Adding a New Rule

Open `gold-layer/gcp/_local_dev.py` and add a new entry to the `RULES` list:

```python
{
    "rule_code": "GKE_HIGH_NODE_COUNT",
    "description": "Cluster has more than 10 nodes with low utilization",
    "recommendation_template": "Consider consolidating workloads — current: {current_sku}",
    "conditions": [
        {"metric": "cpu_utilization_avg", "operator": "lt", "threshold": 30.0},
    ],
    "logic": "AND",
},
```

Then add the corresponding handler in `cr_gke_engine.py` under `_find_candidate()`.

## SQL Configuration (Production)

Located at: `gold-layer/gcp/sql/`

These SQL files are used by the data team to seed the RDS database tables that the production
recommendation engine reads from. They are not needed for local testing — `_local_dev.py`
uses hardcoded rules instead.

### Structure

```
sql/
├── README.md
├── service_configuration/
│   └── cr_gke.sql        -- registers GKE in service_configuration_v2
└── threshold_rules/
    └── cr_gke.sql        -- inserts GKE_IDLE + GKE_OVERPROVISIONED rules
```

### service_configuration_v2

`sql/service_configuration/cr_gke.sql` inserts one row that tells the production engine:

| Column | Value | Description |
|---|---|---|
| service_code | GKE | Unique identifier for this service |
| service_name | Google Kubernetes Engine | Display name |
| cloud_provider | gcp | Cloud provider |
| category | Compute | Service category |
| iceberg_metrics_table | bronze_gcp_metrics_v2 | Iceberg table to read metrics from |
| iceberg_resources_table | bronze_gcp_gke_clusters | Iceberg table to read cluster data from |
| recommender_class | GKEEngine | Python class name to instantiate from rightsize_engine |
| sku_json_file | cr_gke_sku.json | SKU catalog file to load |
| is_active | true | Whether this service is enabled |

### base_finops_threshold_rules_v2

`sql/threshold_rules/cr_gke.sql` inserts two rules:

**GKE_IDLE**

```json
{
  "logic": "AND",
  "conditions": [
    {"metric": "cpu_utilization_avg",    "operator": "lt", "threshold": 5},
    {"metric": "memory_utilization_avg", "operator": "lt", "threshold": 20}
  ]
}
```

Recommendation template: `Idle GKE node pool — delete or scale to 0 nodes (current: {current_sku})`
Savings formula: `total_cost * 1`

**GKE_OVERPROVISIONED**

```json
{
  "logic": "OR",
  "conditions": [
    {"metric": "cpu_utilization_avg",    "operator": "lt", "threshold": 40},
    {"metric": "memory_utilization_avg", "operator": "lt", "threshold": 50}
  ]
}
```

Recommendation template: `Overprovisioned GKE node pool — rightsize from {current_sku} to {target_sku}`
Savings formula: `current_cost - target_cost`

### How to run

```bash
# Run against your RDS instance
psql -h your-rds-host -U username -d database -f sql/service_configuration/cr_gke.sql
psql -h your-rds-host -U username -d database -f sql/threshold_rules/cr_gke.sql
```

Both scripts use a DELETE + INSERT pattern so they are safe to re-run at any time.

### Adding a New Rule in Production

1. Add the new rule to `sql/threshold_rules/cr_gke.sql`
2. Re-run the file against RDS
3. Add the corresponding handler in `cr_gke_engine.py` under `_find_candidate()`
4. Update `_local_dev.py` RULES list to match for local testing

## Troubleshooting

**403 Forbidden on cr_gke_fetch.py**
The service account key is missing or the account lacks `roles/billing.viewer`.
Check the key path or set `GOOGLE_APPLICATION_CREDENTIALS`.

**Machine type not found in catalog**
The machine type used by the cluster is not in the pre-built list in `cr_gke_fetch.py`.
Add it to the `MACHINE_TYPES` dict in `enrich_with_machine_types()` and re-run the fetch script.

**No metrics available for cluster**
The bronze CSV has no metric rows for that cluster. Re-run `local_run_gcp_gke.py` and check
that Cloud Monitoring has data for the cluster in the configured `METRICS_WINDOW_DAYS`.

**Engine returns None**
Either the machine type is not in the SKU catalog, or no cheaper machine type exists in the
same family. Check the logs for `[GKE_OVERPROV]` or `[GKE_IDLE]` messages.
